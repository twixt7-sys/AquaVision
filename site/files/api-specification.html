API Specification

api-specification v2.0.0 updated 2026-07-18 ◎Target raw JSON ↗

Status

Design intent. No API is implemented. Paths and shapes are indicative and will change on contact with a real client.

Design Principles

  • REST over HTTPS, JSON. Boring on purpose. There is nothing in this domain that justifies a novel protocol.
  • One API, three phases. A free Phase 1 farmer and an enterprise Phase 3 operator hit the same base API — tier gates which endpoints return data versus a 402/403, not a different service.
  • Read-heavy. The mobile/web app, the dashboard, and the webpage consuming this repository are the primary clients.
  • Every response that contains a number carries its provenance and, for water-quality data, whether it is a sensor measurement or a self-report. An API that returns bare numbers invites clients to render them as facts.
  • Tenant isolation is enforced at the API boundary, not in the client. A client bug must not be able to expose Farm B's data to Farm A — and, separately, must not be able to leak a private farm record into the public community feed.
  • Uncertainty is never optional in a response. If a client wants a scalar without an error bar, they can discard the field — but we will not omit it.

Response Envelope

Note

Every data response carries this envelope. It is the API-level expression of the repository's provenance discipline, extended to distinguish self-reported app data from calibrated sensor data.

Shape
Data
the payload
Meta
Provenance
verified_external | derived_calculation | modeled_projection | design_target | illustrative_synthetic | assumption
Source Type

sensor_probe | manual_entry | model_inference | farmer_supplied | community_content — present on any response mixing or potentially confusable with water-quality or stock data

Qc Summary
counts by flag, where applicable
Generated Utc
when this response was computed
Staleness
age of the newest underlying measurement or log entry
Completeness

what is missing and why — for a Premium analytics or predictive response, this includes whether the farm's logging history is sparse enough to undermine the result

Warnings
array of human-readable caveats the client SHOULD render
The Warnings Field

Clients are expected to render warnings, not swallow them. A warning like 'this concentration field was calibrated on 3 points with an R² of 0.41' or 'this farm has logged water quality twice in 60 days' is more important than the field itself, and an API that makes it easy to ignore is an API that facilitates overclaiming.

Auth and Tenancy

Model

Per-tenant, where a tenant is a farm. A user may belong to multiple farms with different roles; a farm's data is visible only to that farm's authorised users and to explicitly granted parties (e.g. a booked expert, scoped to the relevant records only).

The Hard Case

The lake-scale carrying capacity use case (UC-10) needs cross-farm aggregation, and the operator who benefits from the lake-wide picture is the one who least wants their own stocking visible in it. The community forum raises a softer version of the same tension for every tier, not just Phase 3.

Position

Cross-tenant reads are not exposed by this API at all. Community content is the one deliberately public surface, and it is explicitly opt-in per post, never a farm's operational data by default. Aggregated lake-scale products are a separate, explicitly-consented pipeline with their own governance. Building a cross-tenant read endpoint 'just for internal use' is how the boundary erodes.

Status
·UNRESOLVED. See Ethics Data Privacy.Json. Flagged Because It Is Cheap To Design Now and Impossible To Retrofit.

What This Api Will Not Do

  • Return a concentration without its uncertainty.
  • Return a model output in the same field as a measurement or a self-report.
  • Return a self-reported water-quality entry with the same confidence badge as a calibrated sensor reading.
  • Accept a dispensing command without a human authoriser.
  • Expose one tenant's data to another, for any reason, including 'research'.
  • Silently include partial profiles in an aggregate.
  • Return specific medication dosing or treatment instructions from the assistant or disease-identification endpoints without routing through a human channel.
  • Move marketplace or consultation money without a traceable order/booking record and its actual applied fee.
  • Provide a person-detection or personnel-tracking endpoint. Not because it is technically hard — because building it would change what this company is.
