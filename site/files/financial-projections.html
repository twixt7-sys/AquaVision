Financial Projections

financial-projections v2.1.0 updated 2026-07-18 ↗Projection ◆Confidence: speculative raw JSON ↗
ℹREAD THIS FIRST

This file has two layers. The first — phase spend plan, milestone-gated funding logic, and scenario structure — describes HOW to think about AquaVision finance without pretending measured traction exists. The second — assumed_financial_projection below — fills in a five-year illustrative model with dollar figures, subscriber counts, and three scenarios (pessimistic / base / optimistic). Every number in that section is tagged assumption or modeled_projection, derived from the inputs in unit-economics.json and pricing-revenue.json, not from company data, because there are zero users and zero revenue today. Use the assumed model to stress-test the plan and show sensitivity; do NOT cite it as forecast, traction, or 'our projections show'. Present the BASE case as the planning case and the pessimistic case unprompted. When Phase 1 launches, replace assumed rows with measured rows — do not amend assumptions in place without updating provenance.

What Can Honestly Be Projected

Statement

Costs are estimable in all three phases. Revenue is only plausibly modelable once Phase 2 is live with real conversion data, and Phase 3 revenue remains as unknowable as in the original hardware-only plan until its own discovery work is done.

Reasoning

We know roughly what a small product team, hosting, and community operations cost (Phase 1). We do NOT yet know Phase 2's conversion rate, ARPU, or churn — see unit-economics.json — so Phase 2 revenue is a scenario, not a projection, until Phase 1 ships and generates real usage data. Phase 3 hardware costs and revenue remain exactly as unknowable as before: no BOM quotes, no price, no customer.

The Honest Note

Dollar envelopes above are order-of-magnitude planning bands (modeled_projection), not signed budgets. Phase 1/2 totals depend on team size and cloud/AI inference costs that Phase 0 discovery will pin down; Phase 3 totals depend on a hardware BOM that still has zero real quotes. The SHAPE — cheap product-first phases that generate their own evidence, followed by an expensive hardware phase that only makes sense once that evidence exists — is the honest and useful part. See assumed_financial_projection for a year-by-year roll-up of these envelopes.

The Milestone Gated Funding Logic

Principle

Raise against de-risking milestones in each phase, not against a revenue projection — for software milestones exactly as much as for hardware milestones.

Why

Each phase gate removes a specific risk: Phase 1 removes 'will anyone use this for free'; Phase 2 removes 'will a viable fraction pay'; Phase 3 removes the same hardware risks the original plan named (payload closes, winch works, profiles validate, farmers pay for automated monitoring). An investor funding this is buying successive de-risking across a software business first and a hardware business second. See funding-ask.json.

The Kill Gates

Phase 1 has an implicit kill gate: if activation and organic retention are poor even at zero price, the free tier is not solving a real problem and Phase 2/3 should not be funded on top of it. Phase 3 keeps its explicit kill criteria from the original plan (payload budget cannot close; commercial farmers show no response capacity or WTP for automated monitoring). A financial plan that includes the conditions under which each phase should NOT proceed is more credible than one that only projects success.

Phase 2 Model When Ready

Monthly recurring revenue = (active free users) x (free-to-paid conversion rate) x (ARPU, blended across subscription + marketplace + consultation) x (1 - monthly churn), compounding month over month across the growth ramp, minus Phase 1/2 cost to serve.

Phase 3 Model When Ready

Unchanged from the original plan: Revenue = (docks deployed) x (sites per dock) x (price per site) x (1 - churn), summed across the deployment ramp, minus cost-to-serve per dock.

The Discipline

When these are filled in, present the BASE case as the plan and show the pessimistic case unprompted, for both phases. A three-scenario model where the pessimistic case is survivable is far more fundable than a single-line projection that only goes up — and this is exactly as true of a freemium user-growth curve as it is of a drone-deployment curve.

What Investors Should Take From This File

Statement

The assumed_financial_projection section exists so reviewers can stress-test the plan — not so the founders can pretend traction exists. Zero users and zero revenue remain the factual starting point.

The Pitch

'Here is our milestone-gated spend plan, here are the assumptions the model rests on — starting with a freemium conversion rate borrowed from generic benchmarks — and here is a three-scenario sensitivity model showing what happens if those assumptions are wrong. Fund Phase 1, watch us replace the assumed rows with measured ones within months, and decide the Phase 3 hardware raise from evidence.'

The Risk

Some investors want a single hockey-stick line and will pass without it. Showing pessimistic/base/optimistic with every input tagged is a filter for investors who diligence assumptions — which is the point.
